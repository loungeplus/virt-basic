
= 既存の仮想マシンの移行

== 導入

このラボのセクションでは、 https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/[Migration Toolkit for Virtualization^]（MTV）を使用して、VMware vSphereからOpenShiftへ仮想マシンをインポートします。この移行ツールキットは、以下の2つのインポート「モード」をサポートしています。

* **コールドマイグレーション（Cold migration）**: 移行を開始する前に、移行元の仮想マシンの電源をオフにします。これがデフォルトの移行タイプです。
* **ウォームマイグレーション（Warm migration）**: 移行元の仮想マシンを実行したままデータをコピーします。データの大部分が移行された後、VMをシャットダウンして最終的なデータを移行先にコピーします。その後、新しいVMを起動できるため、VM内でホストされているアプリケーションのダウンタイムを大幅に短縮できます。

[NOTE]
====
移行ツールキットは、OperatorHubで利用可能なオペレーター（Operator）を使用して、クラスターにすでにデプロイされています。
====

オペレーターのインストールと構成方法に関するドキュメントは、 https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/html/installing_and_using_the_migration_toolkit_for_virtualization/installing-mtv_mtv#installing-the-operator_mtv[こちら^] で確認できます。

組織のニーズに合わせてMigration Toolkit for Virtualizationを構成する方法について詳しく知りたい場合は、以下のリンクにある各プラットフォーム向けのドキュメントを参照してください。

* https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/html/planning_your_migration_to_red_hat_openshift_virtualization/assembly_provider-specific-requirements-for-migration_mtv#ova-prerequisites_mtv[Open Virtual Appliance (OVA)^]
* https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/html/planning_your_migration_to_red_hat_openshift_virtualization/assembly_provider-specific-requirements-for-migration_mtv#cnv-prerequisites_mtv[OpenShift Virtualization^]
* https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/html/planning_your_migration_to_red_hat_openshift_virtualization/assembly_provider-specific-requirements-for-migration_mtv#openstack-prerequisites_mtv[OpenStack^]
* https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/html/planning_your_migration_to_red_hat_openshift_virtualization/assembly_provider-specific-requirements-for-migration_mtv#rhv-prerequisites_mtv[Red Hat Virtualization^]
* https://docs.redhat.com/en/documentation/migration_toolkit_for_virtualization/2.11/html/planning_your_migration_to_red_hat_openshift_virtualization/assembly_provider-specific-requirements-for-migration_mtv#vmware-prerequisites_mtv[VMware vSphere^] 

.*目標*

* VMware vSphere環境の探索
* Migration Toolkit for Virtualization（MTV）に必要な構成の確認
* 移行プラン（Migration Plan）の作成
* OpenShift VirtualizationへのVMの移行
* 組み込みのインタラクティブデモによる高度なオプションの探索

== 移行の前提条件

以下の前提条件がすべての移行に適用されます。

* ISOイメージおよびCD-ROMディスクがアンマウントされていること。
* 各NICにIPv4アドレスまたはIPv6アドレスのいずれかが含まれていること（両方使用していても可）。
* VMのオペレーティングシステムが、OpenShift VirtualizationのゲストOSとして https://access.redhat.com/articles/4234591[認証およびサポート^] されていること。
* VM名には、英小文字（a-z）、数字（0-9）、またはハイフン（-）のみを使用し、最大253文字までであること。最初と最後の文字は英数字である必要があります。名前に英大文字、スペース、ピリオド（.）、または特殊文字を含めることはできません。
* VM名が、OpenShift Virtualization環境内の既存のVM名と重複していないこと。

[NOTE]
====
**Migration Toolkit for Virtualization** は、ルールに準拠していないVMに対して新しい名前を自動的に割り当てます。これにより、移行処理を問題なく続行させることができます。
====

== VMwareからの仮想マシンの移行

OpenShiftに移行するために、3層（3-Tier）アプリケーションがVMware上にデプロイされています。

このアプリケーションは、以下の4つの仮想マシンで構成されています。

* Webサーバーにトラフィックをリダイレクトする1台のHAproxyシステム。
* MariaDBデータベースを実行する1台のLinuxシステム。
* データベースに接続するPHPアプリケーションをホストする、IISを備えた2台のMicrosoft Windowsサーバー。

このラボでは、4つの仮想マシンのうち3つをコールドマイグレーションを使用して移行するタスクを実行します。

[NOTE]
====
HAproxy（ロードバランサー）のVMを移行する必要はありません。OpenShiftは、*Service（サービス）* を使用してSDNに接続されたVMのネットワークトラフィックと負荷分散をネイティブに処理するためです。
====

+

== 高度な機能のインタラクティブデモ

仮想マシンを操作できるようにラボのコールドマイグレーションの完了を待っている間に、この機会を利用して、Migration Toolkit for Virtualizationで実行できるいくつかの追加の高度な移行オプションを紹介したいと思います。このセクションでは、移行プロセス中の仮想マシンの自動再構成や、移行作業中のダウンタイムを短縮するウォームマイグレーションの実行方法について説明します。

この時間を利用して以下のセクションを確認し、インタラクティブデモ画面（英語）をお楽しみください。

=== 自動構成フック（Automated Config Hooks）

自動構成フックは、Migration Toolkit for Virtualizationに組み込まれている、非常に実用的で強力な機能です。

構成フックは、Migration Toolkit for Virtualizationの機能の一部として含まれています。移行プロセスの完了前または完了後に、仮想マシンのプロパティを変更するためのプレ構成フック（Pre-configuration hook）またはポスト構成フック（Post-configuration hook）としてセットアップできます。これらは Ansible 自動化のパワーを活用し、YAML 形式のプレイブックを通じて目的のアクションを実行しますが、これらに個別の https://www.redhat.com/en/hybrid-cloud-solutions/automation?sc_cid=RHCTN0250000435827&gad_source=1&gad_campaignid=20322566154&gbraid=0AAAAADsbVMRTUlnZMtmJPEadK_tiBW92m&gclid=EAIaIQobChMIqt2m1oHxjAMVjnFHAR1rXhMLEAAYASAAEgIdAfD_BwE&gclsrc=aw.ds[Ansible Automation Platform^]. のサブスクリプションは必要ありません。その代わりに、移行プロセス中にターゲットの仮想マシンに対して提供されたプレイブックを実行する、独立した *ansible-runner* ポッドの内部で実行されます。

これらの自動化されたタスクは、移行元のハイパーバイザーで動作していた仮想マシンが、OpenShift Virtualization上でも期待通りに動作することを確認するために非常に価値があります。

ゲスト上のファイルを編集するようなシンプルなタスクから、ハードウェアデバイスの構成を更新するような複雑なタスクまで実行できます。

以下のインタラクティブデモをチェックして、双方の実際の動作例を確認してください。

.デモの手順は以下の通りです。

* リモートの ESXi ホスト上の仮想マシンを検出し、コールドマイグレーションプランを構成します。
* プレ移行フック（Pre-migration hook）を使用して、ネットワークアダプターを VMware ネイティブの形式から OpenShift Virtualization 用の VirtIO 形式に再構成し、移行プロセスの開始タイムスタンプをテキストファイルに書き込みます。
* ポスト移行フック（Post-migration hook）を使用して、ネットワークアダプターの構成をクリーンアップし、別のテキストファイルに移行プロセスの完了タイムスタンプを書き込みます。
* 実行中の仮想マシンが正常に移行され、起動後も行った変更が維持されていることを検証します。

=== ウォームマイグレーション（Warm Migration）

Migration Toolkit for Virtualization のデフォルトの移行タイプはコールドマイグレーションであり、移行期間全体を通じて移行元の仮想マシンの電源をオフにする必要がありますが、多くの組織には、移行作業のために長時間のダウンタイムを許容できない重要なワークロードが存在します。これらのタイプのワークロードに対して、Migration Toolkit for Virtualization はウォームマイグレーション（Warm migration）もサポートしています。

ウォームマイグレーションプロセスは、移行元のハイパーバイザー上で変更ブロック追跡（CBT: Changed Block Tracking）技術を利用することで実現されます。これは、ネットワーク経由で転送されるデータ量を削減するために、バックアップや災害復旧（DR）運用でよく使用される技術です。仮想マシンで CBT が有効になると、VM ディスク上の変更記録を保持するために、一定の間隔でゲストのスナップショットが取得されます。VM を移行する段階になると、ゲストがまだ実行されている間に、最初のデータコピーが転送されます。これにより、マシンが稼働したまま、ダウンタイムなしで大量のマシンデータを転送できます。メンテナンスウィンドウ（システム停止時間）をスケジュールできる時間になったら、Migration Toolkit for Virtualization がカットオーバー（切り替え）をトリガーし、ゲストの電源をオフにして、変更された CBT スナップショットデータのみを転送します。これにより、VM のダウンタイムを大幅に短縮できます。

.デモの手順は以下の通りです。

* 移行元のハイパーバイザー上で実行されている Windows および Linux 仮想マシンを確認します。
* 構成されている Migration Toolkit for Virtualization プロバイダーを確認します。
* カットオーバー（切り替え）フェーズで一時停止するウォームマイグレーションプランを作成し、実行します。

== まとめ

このセクションでは、Migration Toolkit for Virtualizationについて探索し、それを使用して既存の仮想マシンを VMware vSphere 環境から OpenShift Virtualization へ移行する作業を支援しました。また、特定の移行シナリオで活用できるいくつかの高度な機能を示すインタラクティブビデオ（デモ画面）を確認しました。

Migration Toolkit for Virtualization に加えて、現在他に2つの移行ツールキットが利用可能です。これらを組み合わせることで、組織のニーズに応じて、さまざまなタイプのワークロードを OpenShift クラスター内やクラスター間へ移動させることができます。

* https://docs.redhat.com/en/documentation/migration_toolkit_for_applications/8.0/html/installing_the_migration_toolkit_for_applications/intro-to-mta_installing-mta-title[Migration Toolkit for Applications^] - 大規模なアプリケーションのコンテナおよび Kubernetes へのモダナイゼーションへの取り組みを加速させます。

* https://docs.redhat.com/en/documentation/openshift_container_platform/4.21/html/migration_toolkit_for_containers/about-mtc[Migration Toolkit for Containers^] - OpenShift クラスター間でステートフルなアプリケーションワークロードを移行します。

これらの他の移行ツールキットに関する詳細情報については、Red Hat の担当者にお問い合わせください。
