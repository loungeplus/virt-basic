
= 仮想マシンのネットワーク管理

== 導入

このワークショップの前のセクションで触れたように、すべての仮想マシンはデフォルトでOpenShiftのソフトウェア定義ネットワーク（SDN：Podネットワーク）にアタッチされます。これにより、他のVMやOpenShiftネイティブのアプリケーションを含むクラスター上の他のワークロードからのアクセスが可能になり、仮想マシンとそれがホストするアプリケーションをよりモダンなワークフローで一元管理できるようになります。

* SDNは、クラスター内にVMとしてデプロイされているかポッドとしてデプロイされているかを問わず、制御された方法でアプリケーションを抽象化、接続、および外部公開するための追加機能を提供します。これには、別のモジュールで探索するOpenShiftの **Service（サービス）** や **Route（ルート）** 機能が含まれます。
* OpenShiftのネットワークポリシー（Network Policy）エンジンを使用すると、VMユーザーや管理者は、個々のVMまたはプロジェクト／ネームスペース全体に宛てた、あるいはそこから送信されるネットワークトラフィックを許可または拒否するルールを作成できます。

しかし、必要に応じて、仮想マシンを未タグ（ネイティブ）ネットワークやVLANなどの1つ以上の物理ネットワークに直接接続することも可能です。これはSDNに「加えて」利用できる機能であり、例えば管理者が外部IPアドレスからVMに直接接続したり、VM同士がレイヤー2（L2）ネットワークを使用して直接通信したりできるようになります。

ハイレベルな視点で見ると、これはLinuxブリッジなどのホストネットワーキングを構成することによって実現されます。このワークショップのセクションでは、そのプロセスの次のステップとして、VMをそのブリッジに接続し、結果として物理ネットワークに直接接続できるようにするための「ネットワーク接続定義（Network Attachment Definition：NAD）」を作成する手順を説明します。

[NOTE]
====
OpenShift環境では、仮想マシンが接続するためのLinuxブリッジがあらかじめ各コンピュートノード（ワーカーノード）上に構成されているため、外部のネットワークリソースとの間での双方向の接続が容易に行えるようになっています。
====

== ネットワーク接続定義の作成

LinuxブリッジをVMで使用するには、**ネットワーク接続定義（Network Attachment Definition：NAD）** を作成する必要があります。これがOpenShiftに該当ネットワークの存在を知らせ、仮想マシンがそこに接続できるようにするための設定となります。ネットワーク接続定義は、それが作成されたプロジェクトにスコープ（制限）が限定され、そのプロジェクト内にデプロイされた仮想マシンからのみアクセスできます。もしネットワーク接続定義が `default` プロジェクト内で作成された場合は、グローバルに（すべてのプロジェクトから）利用可能になります。これにより、管理者である皆様は、独自のVMの管理権限を持つ特定のユーザーに対して、どのネットワークを利用可能にするか（あるいはさせないか）を制御できます。

[NOTE]
====
ネットワーク接続定義は、OpenShiftに対して既存のネットワークデバイスを使用するように指示します。今回のケースでは、そのデバイスはあらかじめ作成されており、**br-flat** という名前が付いています。必ずこの正確な名前を使用する必要があります。そうしないと、その固有の名前を持つネットワークデバイスが存在するノードをOpenShiftが検出できず、VMをいずれのコンピュートノードにも配置できなくなります。
====

. 左側メニューから **Networking（ネットワーク）**、**Network Attachment Definitions（ネットワーク接続定義）** の順に選択し、**Create network attachment definition（NetworkAttachmentDefinitionの作成）** ボタンをクリックします。
+
image::2026_spring/module-06-networking/06_NetworkAttachDefinition_Create.png[link=self, window=blank, width=100%]

== ユーザー定義ネットワーク

ユーザー定義ネットワーク（User Defined Networks：UDN）が実装される前は、OpenShift Container Platform用のOVN-Kubernetes CNIプラグインは、プライマリ（メイン）ネットワーク上でレイヤー3（L3）トポロジーのみをサポートしていました。Kubernetesの設計原則により、すべてのポッドはメインネットワークにアタッチされ、すべてのポッドがIPアドレスによって相互に通信し、ポッド間のトラフィックはネットワークポリシーに従って制限されます。このコンテナ特有の新しいネットワークアーキテクチャを学ぶことは、従来の多くの仮想化管理者にとって懸念点として挙げられることがよくありました。

UDNの導入により、隔離されたカスタムのレイヤー2（L2）、レイヤー3（L3）、およびローカルネット（LocalNet）のネットワークセグメントが有効化され、デフォルトにおけるKubernetes PodネットワークのL3トポロジーの柔軟性とセグメンテーション機能が向上します。これらのセグメントはすべてデフォルトで相互に隔離されています。これらのセグメントは、デフォルトのOVN-Kubernetes CNIプラグインを使用するコンテナポッドや仮想マシンのプライマリまたはセカンダリネットワークのいずれかとして機能します。UDNは幅広いネットワークアーキテクチャとトポロジーを可能にし、ネットワークの柔軟性、セキュリティ、およびパフォーマンスを向上させます。

クラスター管理者は、UDNを使用して、`ClusterUserDefinedNetwork` カスタムリソース（CR）を活用することにより、クラスターレベルで複数のネームスペースにまたがる追加のネットワークを作成・定義できます。さらに、クラスター管理者またはクラスターユーザーは、`UserDefinedNetwork` CRを使用して、ネームスペースレベルで追加のネットワークを定義することもできます。

ユーザー定義ネットワークには以下のメリットがあります。

* **セキュリティのための強化されたネットワーク隔離** : Red Hat OpenStack Platform（RHOSP）でテナントが隔離されるのと同様に、ネームスペースごとに独自の隔離されたプライマリネットワークを持たせることができます。これにより、テナント間のクロスウェブトラフィックのリスクが軽減され、セキュリティが向上します。
* **ネットワークの柔軟性** : クラスター管理者は、プライマリネットワークをレイヤー2またはレイヤー3のネットワークタイプとして構成できます。これにより、従来のセカンダリネットワークが持っていた柔軟性をプライマリネットワークにもたらすことができます。
* **簡素化されたネットワーク管理** : ユーザー定義ネットワークを使用すると、異なるネットワーク内でワークロードをグループ化するだけで隔離が達成されるため、複雑なネットワークポリシーを策泊する必要がなくなります。
* **高度な機能** : ユーザー定義のネットワーキング機能により、管理者は複数のネームスペースを単一のネットワークに接続したり、異なるネームスペースのセットに対して個別のネットワークを作成したりできます。ユーザーは、異なるネームスペースやクラスター間でIPサブネットを指定して再利用することもできるため、一貫したネットワーキング環境が提供されます。

=== OpenShift Virtualizationにおけるユーザー定義ネットワーク

OpenShift Container PlatformのWebコンソールまたはCLIを使用して、仮想マシン（VM）のプライマリインターフェース上にユーザー定義ネットワーク（UDN）を接続できます。プライマリユーザー定義ネットワークは、指定されたネームスペース内のデフォルトのPodネットワークを「置き換え」ます。Podネットワークとは異なり、プライマリUDNはプロジェクトごとに定義でき、各プロジェクトで固有のサブネットとトポロジーを使用できます。

レイヤー2（L2）トポロジーを使用すると、OVN-Kubernetesはノード間にオーバーレイネットワークを作成します。このオーバーレイネットワークを使用することで、追加の物理ネットワークインフラストラクチャを一切構成することなく、異なるノード上のVM同士を接続できます。

レイヤー2トポロジーは、ライブマイグレーション中もクラスターノード間で永続IPアドレスがそのまま維持されるため、ネットワークアドレス変換（NAT）を必要とせずに、VMのシームレスなマイグレーション（移動）を可能にします。

ただし、プライマリUDNを実装する前に、以下の制限事項を考慮する必要があります。

1. VMへのSSHアクセスを構成するために `virtctl ssh` コマンドを使用することはできません。
2. VMへのポートフォワードを実行するために `oc port-forward` コマンドを使用することはできません。
3. VMへのアクセスにヘッドレスサービス（Headless Service）を使用することはできません。
4. VMのヘルスチェックを構成するためのReadiness（レディネス）プロブやLiveness（ライブネス）プロブを定義することはできません。

=== ユーザー定義ネットワークの操作

ユーザー定義ネットワークを操作する前に、UDNやそれにアクセスする仮想マシンを作成するよりも「前」に、適切なラベルが付与された専用のネームスペースを作成しておく必要があります。VMが存在する既存のネームスペースを新しいネットワークに割り当てたり、既存の通常のネームスペースにUDNを作成したりすることは、OVN-Kubernetesによって拒否されます。

このタスクはクラスター管理者が実行する必要があります。このラボ環境では、適切なラベル（`k8s.ovn.org/primary-user-defined-network`）が付与された **vmexamples-{user}-udn** という名前のネームスペースがあらかじめ割り当てられています。

. **Networking（ネットワーク）** -> **UserDefinedNetworks（ユーザー定義ネットワーク）** の順にクリックし、プロジェクトとして **vmexamples-{user}-udn** が選択されていることを確認して、**Create UserDefinedNetwork（ユーザー定義ネットワークの作成）** ボタンをクリックします。
+

. ドロップダウンリストから現在のプロジェクト **vmexamples-{user}-udn** を選択し、サブネットに `192.168.254.0/24` を追加して、**Create（作成）** を押します。
+
image::2026_spring/module-06-networking/16_UDN_Form.png[link=self, window=blank, width=100%]

== まとめ

このモジュールでは、物理ネットワークの操作と、既存のネットワークに仮想マシン（VM）を直接接続する方法について探索しました。VMを物理ネットワークに直接アタッチすることで、管理者はVMに直接アクセスできるようになると同時に、VMをストレージネットワークや管理ネットワークなどの特化型ネットワークに個別に接続させることができるようになります。

また、クラスター管理者やエンドユーザーに対して、高度にカスタマイズ可能なネットワーク構成オプションを提供し、プライマリおよびセカンダリの双方のネットワークタイプを管理するための遥かに柔軟な体験をもたらす「ユーザー定義ネットワーク（UDN）」についても紹介しました。
